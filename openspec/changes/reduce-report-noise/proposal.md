## Why
- 仓库根目录与 `docs/` 下已经累积了大量 `*_REPORT.md`、`FINAL_*`、`SUMMARY` 等过程性报告文件，且内容高度重复、粒度不一，给 diff/代码审查和日常阅读带来噪音。
- 多数测试/回归脚本在每次执行时都会生成或追加新的报告/日志文件（含 `docs/TEST_REPORT.md`、`logs/regression/*.log`、`full_regression_*.log` 等），这些文件被提交进 Git，导致历史臃肿且难以辨认真正的行为变更。
- 你希望测试报告在每次测试时只作为“本次运行的摘要”存在（例如 PR 描述、CI 工件），不作为长期持久化文件；回归测试计划等规范性文档应长期维护，而报告类文件只在需要时按需生成。
- 目前的 tooling-scripts 规格要求在多个流程中把摘要追加到 `docs/TEST_REPORT.md`，以及持续维护 `docs/scripts_inventory.md`，需要通过一次变更明确区分“长期规范文档”和“按需、默认不持久化的测试报告”。

## What Changes
1. 在 tooling-scripts 规格中新增/修改若干 Requirements：
   - 定义“测试报告持久化策略”：
     - 回归/冒烟/脚本测试默认只输出到 stdout/stderr 或 CI 工件，不再强制写入 `docs/TEST_REPORT.md`。
     - `docs/TEST_REPORT.md` 视为可选/过渡性的人工记录，而非强制持久化目标；推荐在 PR 描述或 CI 注释中附上本次测试摘要。
   - 明确日志与测试输出的仓库策略：
     - `logs/` 下的 `*.log` 一律视为构建产物，不得提交到仓库；需要长期保留的审计信息（如 `logs/reconcile_history.jsonl`）在规格中单独声明。
   - 约束脚本默认不生成新的 Markdown 报告文件：
     - 除 `docs/scripts_inventory.md`（脚本清单）和少数明确列出的规范性文件外，脚本在无显式参数/环境变量时不得创建新的 `*_REPORT.md` / `FINAL_*.md` 等报告文件。
   - 为自动化/AI 助手补充指南：
     - 默认只在对话中用精炼的中文总结测试与脚本执行结果，不主动新建报告/总结类 Markdown 文件；仅在用户明确要求“生成报告文件”时才创建。
2. 为 `docs/scripts_inventory.md` 重新定位：
   - 明确其是“核心指引文档”，须随着脚本变更持续维护更新。
   - 其他临时性/一次性脚本分析报告（如某轮故障回顾）建议迁移到 `docs/history/` 或在后续清理中删除。
3. 为日志与测试报告输出建立统一模式：
   - 回归/冒烟测试脚本通过参数（如 `--report` 或环境变量）显式开启额外 Markdown 报告输出，默认仅写 stdout + 非提交的 `logs/*.log`。
   - 规格中为这些参数约定命名与行为边界，为后续脚本改造提供统一约束。

## Impact
- tooling-scripts 规格（特别是当前要求追加到 `docs/TEST_REPORT.md` 的几处）需要更新，以配合“测试报告不再持久化”的方向。
- 回归/冒烟/验证脚本的行为将被规范为“默认仅输出到 stdout 与未跟踪日志文件”；CI 与开发者需要改为在 PR 描述或 CI 注释中附上测试摘要，而不是依赖新生成的报告文件。
- 现有已经提交的日志文件（`logs/regression/*.log`、`webui_e2e*.log` 等）和大量过程性报告文件将作为后续实现任务，按“删除 / 迁移到 docs/history / 保留极少数规范性文档”的策略逐步清理。
- AI/自动化助手在本仓库中的使用方式将更加“轻量”：以对话形式报告要点，只有在你明确要求生成报告文件时才写入新的 Markdown。
